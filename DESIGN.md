# 資治通鑑 翻訳プロジェクト — 設計仕様 (v1)

> 本書は grilling セッション(2026-06-20)で確定した設計判断の正本。
> 仕様変更はこのファイルへの差分で管理する。

## 1. 目的とスコープ
- **対象**: 資治通鑑 全294巻・本文約300万字を全訳。
- **成果物**: 完全な逐語訳ではなく、**全編を平易な現代日本語で読めるナレッジベース**。
- **訳文スタイル**: 読みやすい口語超訳。原典に正確であることを前提に、現代人の理解を助ける脚色は許容。

## 2. 品質契約(脚色の線引き)
- **許される脚色**: 主語の補完 / 語順の整理 / 比喩の言い換え / 読みやすい段落分け。
- **許されない脚色(差し戻し対象)**: 原文にない事実・因果・心理描写の追加 / 年月日・数字の改変 / 発言の創作。
- **許容誤差** は「許される脚色」と同項目。「許されない脚色」に該当したら差し戻し。
- 新規の判定が必要なケースは **都度ユーザーに質問** して判断を仰ぐ。

## 3. エンジンとオーケストレーション
- **翻訳エージェント = Claude / オーケストレーター = Claude Code (Pro サブスク上)**。
- **検証 = 別ベンダー別モデル (GPT)**。Codex CLI をサブプロセス / MCP で駆動して自動化。サブスク内で完結。
- Codex との往復は **JSON / ファイルで厳格にスキーマ化**(CLI テキスト I/O のパース破綻を防ぐ)。
- 各サブスクのレート上限が律速。**長期運用前提** として許容。
- **コスト方針**: 調査・検証など委譲可能なタスクは積極的に Codex 側へ寄せ、Claude / Codex 双方の利用量を最適化する(従量 API は使わない)。
- **調査委譲の作法**(検証①の運用知見): Codex のネイティブ web 検索は headless `exec` 下で Cloudflare 403 に阻まれるため、調査委譲時は「`curl` / HTTP で直接取得」を明示指示する(サンドボックスはネットアクセス許可で実行)。
- **オーケストレーション機構(検証②で確定)**: `codex exec` をサブプロセス起動。入力=stdin(プロンプト)+ ファイル(出典/コンテキスト)、出力=`--output-schema` で JSON 強制 + `--output-last-message` で捕捉してパース。`codex exec` は毎回新規セッションのため §4「ラウンド独立セッション」を無コストで満たす(`resume` 不使用)。詳細 `research/02-orchestration.md`。
- **Codex レート予算ガード**(運用ルール): Codex は ChatGPT サブスクの **5 時間レート上限が律速**。残量が乏しいときは新規 Codex タスクを投げない。構築・スクリプト作成など Codex 不要の作業は **Claude 側に寄せる**。Codex は「別モデル独立レビュー」など Codex でしかできない工程に温存する。実走時はレビューをバッチ化し、難所以外は `model_reasoning_effort` を下げる。
- **実行系 3 種(2026-06-20)**:
  1. **Claude(本体)** = オーケストレーター + 翻訳。Claude サブスク。
  2. **Codex / GPT**(`codex exec`)= 独立クロスベンダー・レビュー(品質クリティカル)。ChatGPT サブスク、5h レート律速。
  3. **agy / Gemini**(`agy -p "…" --dangerously-skip-permissions [--add-dir DIR]`、v1.0.10)= ワンショット実装・厳密指示下の検証タスク。精度はやや劣るため品質クリティカルな翻訳レビューには使わず、コード生成・機械的検証(スキーマ/形式チェック・ソース間差分・NER 下処理)や Codex 前の安価な一次スクリーニングに用い、Claude/Codex 予算を節約する。
- **使い分け原則**: 検証可能で厳密指示にできる作業 → **agy** 優先(最安)。クロスベンダーの判断品質が要る独立レビュー → **Codex**。設計判断・オーケストレーション・翻訳本体 → **Claude**。

## 4. レビューループ
- レビュアーは **「誤りがある前提」** でチェック。**独立セッション** で実行。
- レビュアーへ渡す出典 = **本文原文 + 胡三省注**(根拠集合のいずれにも無い情報のみ「捏造」と判定)。
- 注由来の挿入は **`〔注:…〕` でマーキング**。レビュアーは捏造判定から除外、読者にも由来を明示。
- フロー: 翻訳 → レビュー → 誤り指摘を翻訳 AI へ FB → 修正 → **前回とは別の独立セッション** のレビュアーが再チェック。
- **合格条件**: 「誤りがある前提」でチェックしても誤りが見つからない状態。
- **最大反復 = 3**。未収束チャンク、およびラウンド間でレビュー指摘が **矛盾** したチャンクは、**停止してユーザーにアラート**。

## 5. チャンク / ファイル設計
- **ファイル = 年**(KB レコード単位・Git 管理単位)。巻ごとにディレクトリ(294)。
- **翻訳の作業単位 = 原文 1,500〜2,500 字**(ファイル単位とは分離)。
  - 範囲内の年はそのまま 1 チャンク。
  - 超える年は順序付きサブチャンクに分割 → 年ファイルへ結合。
  - 短い年は結合しない(ファイル = 年の清潔さを守る)。
- 各作業チャンクに **コンテキストパケット** を同梱:
  - 全体内の位置づけ(巻 / 年 / 前後要約)
  - 関連人物一覧(辞書由来)
  - 直前の訳出セグメント
- **実装状況(2026-06-20)**: `pipeline/segment.py` で全294巻を年エントリに分割成功(**失敗 0 巻**)。
  - 年マーカーは **4 形式**に対応: 裸の漢数字年(周/漢)、元号+年(晋以降)、`==元号N年==` 見出し(唐)、`載`(天寶年間)。
  - 総年エントリ **1,397**(= 基本翻訳単位の概数)。本文 2,500字超 462件・5,000字超 144件 → サブチャンク分割が必要(設計どおり)。
  - 胡注は `{{*|…}}` を抽出し本文に `⟦nK⟧` で位置保持(note_anchor_offset)。臣光曰を含む年 105。
  - `pipeline/build_staging_kb.py`: 全294巻 → 年レコード1,397 → **作業チャンク2,096**(平均1,501字 / 最大2,501)。注 resolve・メタデータ・サブチャンク化済み。出力 `data/staging/kb/卷NNN.json`(gitignore)。
  - `pipeline/western_years.py`: **巻レベル西暦を干支から決定論的に算出**(LLM不使用)。起403BCE・終959CE 一致、盡干支ミスマッチ0、補間38巻・境界異常5(`manifests/volume_years.json` に記録)。staging KB に注入済み。
  - 残 TODO: 巻内の年単位西暦(在位表が必要)、後世校勘レイヤ判別、Kanripo クロスチェック、エンティティ辞書付与(§6 C)。

## 6. エンティティ辞書
- **人物・地名・官職の表記揺れ(本名 / 字 / 諡 / 官職名)→ 正規名マッピング**。
- 初期辞書は **なるべく正確に調査して初期作成**。
- **依存インデックス**: エントリ → 参照チャンクの逆引きを保持。
- 辞書更新時は **参照チャンクのみ** 遡及再翻訳・再レビュー(全コーパス再処理はしない)。Git 差分で監査。
- **初期 seed ソース(検証③ 2026-06-20 で確定)**:
  - 人物: **CBDB SQLite**(`BIOG_MAIN` + `ALTNAME_DATA` + `ALTNAME_CODES`)。唐は高密度(約5.8万)、五代/南北朝/隋は有用、**戦国〜三国は疎 → 本文 NER で補完必須**。
  - 地名: **TGAZ + CHGIS**(`name+yr` API / CSV / MySQL dump)。秦漢以後は有用、戦国前半は疎。
  - 官職: **CBDB `OFFICE_CODES` + `POSTED_TO_OFFICE_DATA`**。唐に強い、漢〜隋・五代は本文抽出で補完。**王朝で職掌が変わるため「表記→官職ID」平坦化は禁止**(`office_name_instance` と `office_concept` の 2 層)。
  - 補完: Wikidata(CC0)、DILA(仏教人物・地名)。Hucker 作業版は権利未確認 → 内部照合専用。
- **同定は候補集合として扱う**: 同じ字/諡/官職名/地名が複数を指すため、表記→単一正規名にしない。王朝・生没/存続年・チャンク内共起で絞る。
- **辞書スキーマ**: entity / names / authority_ids / source_assertions(典拠・version・**license をレコード単位で保持**)/ place_detail / office_detail + 検索用 `name_index`。詳細 `research/03-entity-dictionary-sources.md`。
- **実装状況(2026-06-21, `pipeline/build_dict.py`)**: seed 構築済み。**人物 75,407(異名17,012)・官職 34,062・地名 13,636(対象時代)**、`name_index` **120,928 表記(曖昧17,730=候補集合の妥当性)**。出力 `dict/*.jsonl`(gitignore)+ `dict/_manifest.json`。
  - 人物・官職は **CBDB 繁体字=本文と一致**し照合可(諸葛亮/曹操/太守 で確認)。era フィルタ動作(王安石=宋代を正しく除外)。
  - 地名(TGAZ)は**簡体字**。型接尾辞除去で簡=繁の地名は照合可(魏/蜀/邺)。繁体字差分(洛陽/長安 等)の照合は **opencc s2t が必要 → 当環境は pip 不可で TODO**(コードは s2t-ready)。
  - 残 TODO: 地名 s2t、年情報の無い人物の本文 NER provisional 化、辞書↔チャンクの依存インデックス連携。

## 7. 胡三省注ポリシー
- **基本 B**(内部参照のみ、出力には出さない)。
- 読みやすさ・理解に寄与する場合のみ、**許容脚色として本文に溶かし込む** + `〔注:…〕` マーキング。
- レビュアーには常に注を出典として供給(§4 と整合)。
- **内容レイヤの区別**(検証①で判明): 本プロジェクトの「注」は胡三省注を指す。司馬光自身の論賛「臣光曰」は注ではなく本文系として扱い、後世校勘記は翻訳対象外(メタとして保持)。レイヤ判定をパイプライン前段(取り込み正規化)で行う。

## 8. KB スキーマ
- 最小単位 = 年(= ファイル)。
- 年レコードのメタデータ: **巻番号 / 年号(元号) / 西暦(年単位のみ) / 登場人物リスト / 地名リスト / 原文位置(出典・revision・ページ)**。
- セグメント単位の最低保持項目(検証①の推奨): `juan, segment_id, main_text, hu_note[], note_anchor_offset, source, source_revision, source_page, license, normalization_log`。
- **内容レイヤを潰さず別フィールドで保持**: ① 司馬光本文 / ② 司馬光論賛「臣光曰」/ ③ 胡三省注 / ④ 後世校勘記。翻訳・脚色ポリシーはレイヤごとに適用。
- **干支日付 → 西暦の細密変換はスコープ外**(数字改変リスク回避)。原文の年号・干支は保持。

- **原文層 (source of record)**: **Kanripo `KR2b0007`**(CC BY-SA 4.0)。commit `80174f61e491db29f9921d0d3e54a59649419aa9` に固定。`_001`〜`_294` のみ採用(`_295` は空スタブ・除外)。原字・胡注(小字括弧)・`<pb:>` ページ座標を保存。OCR 誤りが実在(例: 巻156 標題「吳三省」)。

## 10. ライセンス整理(要・公開方針決定)
公開する KB / 辞書に関わるソースのライセンスが層ごとに異なり、特に CBDB の **非商用 (NC)** 条項が効く。公開フェーズ前に方針確定が必要。

| 層 | ソース | ライセンス | 含意 |
|---|---|---|---|
| 原文 / 注 | Kanripo, 維基文庫 | CC BY-SA 4.0 | 帰属 + 継承(ShareAlike) |
| 人物 / 官職 | CBDB | CC BY-NC-SA 4.0 | 帰属 + **非商用** + 継承 |
| 地名 | CHGIS / TGAZ | 表示が矛盾(EULA 非商用 / API CC BY-NC / GitHub GPL / Dataverse CC0) | 内部利用可、再配布は要書面確認 |
| 補完 | Wikidata | CC0 | 制約なし(最も自由) |
| 検証 | NCL 影印 | public domain | 制約なし |

- **全ソースで provenance(出典・version・license)をレコード単位に保持**(辞書スキーマの `source_assertions`、セグメントの `license` フィールド)。これによりライセンス別 export と後からの方針変更が可能。
- **決定済み(2026-06-20): 非商用方針**。CBDB / CHGIS / 維基文庫を成果物に埋め込み可。成果物は **CC BY-NC-SA 系(帰属 + 非商用 + 継承)** で公開する前提。各ソースの帰属表示を成果物に明記する。詳細 `research/03-entity-dictionary-sources.md`。
